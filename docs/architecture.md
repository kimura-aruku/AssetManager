# AssetManager 設計仕様

## 文書情報

- **状態**: 承認済み
- **承認日**: 2026-07-19
- **前提**: `docs/specification.md` は承認済み
- **最終更新日**: 2026-07-19

## 1. 設計目標

- Windows向けの導入しやすいデスクトップアプリケーションとする。
- すべてのレコードで共通するカラムを、後から柔軟に追加・変更できるようにする。
- SQLiteや外部データベースのセットアップを不要にする。
- 素材の実ファイルと、アプリが保持する管理情報を明確に分離する。
- 1レコードの破損が他レコードへ波及しにくい保存形式にする。
- 数十から数千レコードを実用的な速度で扱う。
- UIテーマを将来差し替えられる構造にする。
- 公開リポジトリへユーザーデータやローカル固有情報を混入させない。

## 2. システム境界

AssetManagerが管理するものは以下とする。

- 素材レコードとカラム値
- 共通カラム定義
- 種類、タグ、タグ分類、購入／入手元マスタ
- 検索条件と一覧表示設定
- パスの存在確認結果
- アプリ設定
- エラーログ

AssetManagerが管理しないものは以下とする。

- 素材の実ファイルそのもの
- 実ファイルのコピー、移動、削除
- クラウドやネットワーク上のデータ
- JSONファイルの外部編集との同期

## 3. 全体構成

技術スタックに依存せず、責務を次の4層に分ける。

| 層 | 主な責務 |
|---|---|
| UI | 3ペイン画面、表、編集、検索、状態アイコン、ダイアログ |
| Application | レコード操作、一括編集、検索、パス確認、設定保存の処理手順 |
| Domain | レコード、カラム定義、値の型、中央固定カラム、種類、タグ、ライセンス状態のルール |
| Infrastructure | JSON永続化、ファイル存在確認、Windows Explorer連携、ログ |

## 4. 動的カラムモデル

### 4.1 基本方針

- すべてのレコードは同じカラム定義セットを使用する。
- カラム定義には変更されない一意なIDを付ける。
- レコードJSONは、カラム名ではなくカラムIDに対して値を保存する。
- 表示名を変更しても、既存レコードの値との対応が失われないようにする。
- 標準カラムとカスタムカラムを同じ定義形式で管理する。
- アプリ固有機能が必要な標準カラムには、表示名とは別に固定のシステム役割を持たせる。
- 標準カラムを非表示にしても、保存済みの値は削除しない。

### 4.2 カラム定義

- `systemRole` は、素材名、対象パス、ライセンス最終確認日など、アプリの挙動に必要な標準カラムだけに設定する。
- 中央固定カラムは、非表示、名称変更、型変更、削除を禁止する。
- 詳細情報用の標準カラムは表示／非表示だけを変更できる。
- カスタムカラムは名称、型、表示／非表示、削除を変更できる。
- 詳細情報用のカラムは、任意に中央ペインへ追加できる。
- お気に入りは中央ペインの一番左、検索条件と詳細ペインの一番上へ固定表示する。
- 対象パス列には行の対象パスをエクスプローラーで表示するボタンを設ける。
- ライセンス期限は動的カラムより後ろの一番右へ固定表示する。
- 並べ替えとグループ分けは実装しない。

正式なJSON構造は「カラム定義とレコードJSON形式」で定義する。

### 4.3 カラム型

最低限、以下の型を用意する。

- `text`
- `multilineText`
- `number`
- `date`
- `boolean`
- `url`
- `singleSelect`
- `multiSelect`
- `filePath`
- `folderPath`

カスタムカラムには、ユーザーが変更できる汎用的な複数値設定を設けない。複数の制作者と販売者は文字列配列、関連文書と関連URLはタイトルと値を持つオブジェクト配列として、標準カラム固有の構造で表現する。

## 5. JSON保存方式

### 5.1 論理配置

```text
<データルート>/
├─ manifest.json
├─ records/
│  ├─ <record-id>.json
│  └─ <record-id>.json
├─ definitions/
│  ├─ fields.json
│  ├─ asset-types.json
│  └─ tags.json
├─ settings.json
├─ views.json
├─ undo/
└─ transactions/
```

データルートのデフォルトは `%LOCALAPPDATA%\kimura-aruku\AssetManager\Data` とし、設定から変更可能にする。固定のアプリ領域と変更可能なデータルートの分離は「配布・初回起動・アプリデータ作成フロー」で定義する。

### 5.2 レコード形式

正式なJSON構造は「カラム定義とレコードJSON形式」で定義する。`id`、`schemaVersion`、`createdAt`、`updatedAt` はカラム値とは分け、アプリが管理するメタデータとする。

「現在のバージョン」と「素材最終更新日」は、それぞれ独立した単一値カラムとして扱う。購入または取得の時点は「購入／入手日」で管理するため、ダウンロード日カラムは設けない。レコードメタデータの `updatedAt` はアプリ上でレコードを更新した日時であり、「素材最終更新日」とは区別する。

### 5.3 JSON採用理由

- データベースサーバーやSQLiteファイルの準備が不要である。
- 初回起動時にアプリが必要なフォルダーと定義ファイルを自動生成できる。
- カラム追加時、既存レコードには値が存在しない状態として扱える。
- 1レコードだけを書き換えられる。
- 破損の影響を1レコードに限定しやすい。
- バックアップや障害調査が比較的容易である。

JSONは内部保存形式であり、ユーザーによる直接編集は前提にしない。JSONの柔軟性だけに依存せず、カラム定義とスキーマバージョンで整合性を保つ。

### 5.4 保存の安全性

- 更新内容を同一ボリューム上の一時ファイルへ完全に書き込む。
- JSONとして読み戻せることを確認する。
- 一時ファイルを対象ファイルと置き換える。
- 保存失敗時は既存ファイルを維持する。
- 各レコードに `schemaVersion` を保存する。
- データ形式変更時は、読み込み時に現在のモデルへ変換する。

## 6. パス管理

- 対象パスと補助パスは文字列としてJSONへ保存する。
- 対象パスは1レコードにつき1件とする。
- Windowsの絶対ローカルパスだけを正式対応とする。
- ファイル／フォルダー選択時と保存時にパス形式を検証する。
- 大文字小文字、末尾区切り、`.`、`..` などを考慮した比較用パスを生成する。
- 表示用にはユーザーが登録したパスを保持する。
- 対象パスの重複判定は比較用パスで行う。
- 補助パスには重複制約を設けない。
- 存在しないパスも値として保持し、リンク切れ状態を別途表示する。
- 対象パスが空欄の場合もレコードを保存し、未登録警告を表示する。

### 6.1 存在確認

- 起動時確認は設定で無効化できる。
- 手動確認はいつでも実行できる。
- 数千件のパス確認でUIを停止させないよう、非同期かつ同時実行数を制限して処理する。
- 確認結果はレコードの永続データとは分離し、セッション状態または再生成可能なキャッシュとして扱う。
- 初期版ではOSのファイル変更通知を購読しない。

## 7. ファイル監視を採用しない理由

Windowsでは `FileSystemWatcher` 相当の仕組みにより、フォルダー内の作成、変更、削除、名前変更を監視できる。ただし、本アプリでは対象パスが多数のフォルダーへ分散するため、以下の問題がある。

- 監視対象フォルダー数が増える。
- アプリを終了している間の変更は検出できない。
- 大量変更時に通知が欠落する可能性がある。
- 一部アプリの保存処理では複数の通知が発生する。
- ドライブの一時切断と削除を区別する追加設計が必要になる。

起動時確認と手動確認で要件を満たすため、常時監視は実装対象にも将来機能にも含めない。

## 8. 一覧・検索

- 初期版では永続検索インデックスとメモリ内の専用検索インデックスを作らない。
- 読み込み済みのレコードモデルが持つ各カラム値へ、カラム単位で検索条件を評価する。
- 初期版では表に最初の100件だけを表示する。
- 「さらに読み込む」操作ごとに次の100件を表示する。
- 100件という単位を設定可能にする機能は将来追加する。
- 表示件数の制限と、検索対象データの読み込み範囲は分離する。
- 検索は未表示のレコードも含む全レコードを対象とする。
- 全カラム横断検索は提供せず、検索条件の入力UIを左ペインへ集約する。
- 複数カラムの検索条件はANDで結合する。
- テキスト検索では検索語とカラム値をUnicode正規化形式KC（NFKC）へ正規化し、`OrdinalIgnoreCase` 相当で比較することで、大文字／小文字と全角／半角を区別しない。
- テキスト検索は部分一致とし、空白区切りの検索語は分割して、すべての語が同じカラム値に含まれる場合に一致とする。
- カラム名は検索対象に含めない。
- チェック項目は文字列化せず、カラムの検索条件でオン／オフを直接評価する。
- 種類、タグ、単一選択、複数選択で複数候補を指定した場合は、候補をORで結合する。
- パスカラムは表示用のパス文字列に対して部分一致検索する。
- 初期版では空欄／空欄以外、数値の大小・範囲、日付の前後・期間などの高度な演算子を提供しない。
- 検索速度の固定閾値は設けない。

## 9. 表編集

- 行、列、セル範囲の選択状態を表現できるUIコンポーネントを採用する。
- 複数行を選択する場合、各行で選択されるカラム範囲を一致させる。
- 行ごとに異なるカラム範囲を同時選択する不連続な選択は許可しない。
- `Ctrl`操作による非連続行の選択を許可するが、選択カラム範囲は全行で同一にする。
- 行ヘッダーはレコード全体、カラムヘッダーは現在読み込み済みの全行にある同一カラムを選択する。未読み込みの検索結果は選択対象にしない。
- 複数行のセルを編集した場合は、実際に編集したカラムの値だけを選択行へ一括適用する。
- コピー内容はタブ区切りのカラムと改行区切りの行を持つ矩形データとして扱う。
- コピー元が1行の場合、同じカラム範囲を選択した貼り付け先の各行へコピー元の1行を繰り返し適用する。
- コピー元が複数行の場合、貼り付け先と行数・カラム数が一致するときだけ行列を維持して適用する。不一致時は変更せずエラーを表示する。
- コピーする値と貼り付け先の型を全件検証する。対象パスへの貼り付けは拒否する。
- 一括編集と貼り付けは `transactions/<transaction-id>/` を使用し、全件成功または全件ロールバックとする。
- 一括操作全体をアンドゥ／リドゥ履歴上の1操作として記録する。

## 10. 状態判定

レコードの状態アイコンは、保存済みの状態と派生状態を組み合わせて決定する。

派生状態の例は以下とする。

- 対象パスまたは重要な補助パスのリンク切れ
- ライセンス期限切れ
- ライセンス最終確認日が警告日数を超過
- 対象パスが未入力
- 条件不明
- 要再確認

- 該当する状態アイコンは省略せず、すべて表示する。
- アイコンはエラー、警告、情報の重大度順に並べ、同じ重大度内では状態IDによる固定順とする。
- アイコンへマウスを合わせると、状態の名称と判定理由をツールチップで表示する。
- 複数状態を単一アイコンへ集約しないため、状態を隠すための優先順位は設けない。

中央ペインのライセンス条件は、許可項目と必須項目を複数の小さなアイコンで表示する。

## 11. Windows連携

Infrastructure層へ次の機能を分離する。

- ファイル／フォルダー選択ダイアログ
- ドラッグ＆ドロップされたパスの受け取り
- 既定アプリで開く
- エクスプローラーで表示する
- ファイルの場合にエクスプローラーで選択する
- パスの存在確認

実ファイルを変更するAPIは初期版では持たない。

## 12. ログ

- エラーおよび障害調査に必要な情報をローカルファイルへ記録する。
- ローカルパスはマスクしない。
- 日単位で `AssetManager-yyyyMMdd.log` を作成する。
- 1ファイルが10MBへ到達した場合は連番を付けた同日ログへ切り替える。
- 起動時に7日を経過したログを削除する。
- 外部への自動送信処理は実装しない。

## 13. UIテーマ

- 色、余白、フォント、コントロールスタイルをテーマ資源へ集約する。
- 初期版では通常テーマだけを提供する。
- 画面ロジックが特定テーマの色へ依存しないようにする。
- 将来ライト／ダークテーマを追加できる構造にする。

## 14. アンドゥ／リドゥ履歴

- レコード編集、一括編集、貼り付け、レコード削除、カスタムカラム削除を共通のコマンド履歴で管理する。
- 操作ごとに更新前と更新後の情報を `undo/<session-id>/` へ保持し、アンドゥとリドゥの両方向へ適用できるようにする。
- 複数ファイルを変更する操作のアンドゥ／リドゥにもトランザクション方式を使用する。
- レコード削除では素材の実ファイルを変更せず、削除したレコードJSONを履歴の復元情報として保持する。
- アンドゥ後に新しい操作を確定した場合は、リドゥ側の履歴を破棄する。
- アプリの正常終了時にそのセッションの履歴を削除する。
- 異常終了で履歴が残った場合は次回起動時に自動削除し、前回セッションのアンドゥ／リドゥ履歴は復元しない。
- アンドゥ／リドゥの適用前に現在のデータを検証し、対象パス重複などの制約違反がある場合は履歴位置を変更せずエラーを表示する。
- 管理情報全体のバックアップ／復元は将来機能とする。

## 15. テスト方針

- テストフレームワークはxUnitを使用する。
- カラム定義と各型の値検証を単体テストする。
- 中央固定カラム、対象パス重複、種類・タグの参照整合性を単体テストする。
- 検索条件、一括編集、アンドゥ／リドゥ、ViewModelの画面非依存ロジックを単体テストする。
- JSONの保存、読み込み、原子的置換、旧スキーマ変換を単体テストする。
- 一時ディレクトリを使い、パス存在確認とWindows連携の境界を結合テストする。
- 数千件のレコードとパスを生成して、起動、検索、100件追加表示を性能確認する。
- ローカルパス、リンク切れ、権限不足、不正JSONを異常系として確認する。
- WPF画面の見た目、マウス操作、キーボード操作、操作感は初期版では自動化せず、受け入れ手順に基づいて手動確認する。

## 16. 採用技術スタック

| 項目 | 採用技術 |
|---|---|
| 言語 | C# |
| ランタイム | .NET 10 LTS |
| UI | WPF / XAML |
| UI構成 | MVVM |
| 一覧 | WPF標準 `DataGrid` の拡張 |
| JSON | `System.Text.Json` |
| 対象 | Windows x64 |
| 配布 | 自己完結型のフォルダーをポータブルZIPとして配布 |

WPFを選定した理由は以下とする。

- Windows専用という仕様に一致する。
- 標準 `DataGrid` が行、セル、複数選択、編集、動的な列生成を提供する。
- Windows Explorerやファイル選択ダイアログとの連携が容易である。
- Windows App SDKランタイムを追加で配布せず、自己完結型の.NETアプリとして配布できる。
- XAMLのリソースを分離することで、将来のテーマ切り替えを実装できる。
- クロスプラットフォーム対応用の追加フレームワークを必要としない。

## 17. 配布・初回起動・アプリデータ作成フロー

### 17.1 配布構成

- `win-x64` 向けに自己完結形式で発行する。
- 発行フォルダー全体をZIP化する。
- ユーザーは任意の場所へZIPを展開し、`AssetManager.exe` を実行する。
- .NETランタイムの事前インストール、インストーラー、管理者権限を要求しない。
- MVPでは単一EXE化せず、診断しやすい通常の発行フォルダーを使用する。
- アプリは展開先フォルダーへユーザーデータを書き込まない。

### 17.2 固定アプリ領域とデータルート

```text
%LOCALAPPDATA%/kimura-aruku/AssetManager/
├─ bootstrap.json
├─ logs/
└─ Data/                 # デフォルトのデータルート
   ├─ records/
   ├─ definitions/
   ├─ undo/
   ├─ transactions/
   ├─ manifest.json
   ├─ settings.json
   └─ views.json
```

- `%LOCALAPPDATA%\kimura-aruku\AssetManager` は固定のアプリ領域とする。
- `bootstrap.json` は現在使用するデータルートの絶対パスを保持する。
- `logs` はデータルートを変更しても固定アプリ領域に残す。
- 素材レコード、カラム定義、種類、タグ、表示設定はデータルートへ保存する。
- データルートのデフォルトは固定アプリ領域内の `Data` とする。
- データルートは設定画面から変更できる。

### 17.3 初回起動

初回起動時はセットアップウィザードを表示せず、次の処理を自動実行する。

1. 固定アプリ領域とログフォルダーを作成する。
2. `bootstrap.json` がなければ、デフォルトのデータルートを設定して作成する。
3. データルートと必要なサブフォルダーを作成する。
4. 標準カラム定義、種類のプリセット、空のタグ定義、初期設定を作成する。
5. JSONとスキーマバージョンを検証する。
6. 起動時パス確認の設定に従って存在確認を開始する。
7. メイン画面を表示する。

既存ファイルは上書きせず、不足している初期ファイルだけを作成する。アプリ領域またはデータルートを作成できない場合は、エラー内容を表示し、別のデータルートを選択するか終了できるようにする。

### 17.4 データルート変更

- 初期版は1つのデータセットだけを扱う。
- 変更先には空のフォルダーを指定する。
- 現在のデータを変更先へコピーし、全ファイルの検証後に `bootstrap.json` を更新する。
- 更新完了前は元データを削除しない。
- 正常に切り替えた後、元データを削除するか残すかを確認する。
- 既存データセットへの切り替えは、将来の複数データセット機能として扱う。

### 17.5 多重起動

JSONへの同時書き込みを防ぐため、アプリはユーザー単位で1インスタンスだけ起動可能にする。2回目の起動時は、既存ウィンドウを前面へ表示して新しいプロセスを終了する。

## 18. カラム定義とレコードJSON形式

### 18.1 共通規則

- JSONはUTF-8、BOMなし、プロパティ名はcamelCaseで保存する。
- 各JSONルートに `schemaVersion` を持たせる。
- レコードIDとカスタムカラムIDにはUUID Version 7を使用する。
- 組み込みカラムIDは `builtin.name` のような固定文字列とする。
- カスタムカラムIDは `custom.<UUID>` とする。
- カラムの表示名を変更してもIDは変更しない。
- 日付だけを表す値はJSON内部で `YYYY-MM-DD` とし、画面では `YYYY/MM/DD` と表示する。
- 登録日時・更新日時はUTCのISO 8601形式とし、末尾に `Z` を付ける。
- WindowsパスはJSON文字列として保存し、アプリ内では絶対パスとして検証する。

### 18.2 カラム定義ファイル

`definitions/fields.json` は、すべてのレコードで共有するカラム定義を保持する。

```json
{
  "schemaVersion": 1,
  "fields": [
    {
      "id": "builtin.name",
      "origin": "builtIn",
      "systemRole": "recordName",
      "label": "素材名",
      "type": "text",
      "mainTableVisible": true,
      "mainTableRequired": true,
      "detailVisible": true,
      "userCanHide": false,
      "userCanRename": false,
      "userCanChangeType": false,
      "userCanDelete": false
    },
    {
      "id": "custom.019f0000-0000-7000-8000-000000000001",
      "origin": "custom",
      "systemRole": null,
      "label": "自由記入欄",
      "type": "multilineText",
      "mainTableVisible": false,
      "mainTableRequired": false,
      "detailVisible": true,
      "userCanHide": true,
      "userCanRename": true,
      "userCanChangeType": true,
      "userCanDelete": true
    }
  ]
}
```

配列内の順序は標準定義順とカスタムカラムの作成順を表す。ユーザーによる並べ替え機能は提供しない。

### 18.3 カラム型

ユーザーがカスタムカラムへ指定できる型は以下とする。

| 型 | JSON値 |
|---|---|
| `text` | 文字列 |
| `multilineText` | 文字列 |
| `number` | 数値 |
| `date` | `YYYY-MM-DD`形式の文字列 |
| `boolean` | `true`または`false` |
| `url` | URL文字列 |
| `singleSelect` | 選択肢ID文字列 |
| `multiSelect` | 選択肢ID文字列の配列 |
| `filePath` | 絶対ファイルパス文字列 |
| `folderPath` | 絶対フォルダーパス文字列 |

以下は組み込みカラムだけが使用する専用型とする。

| 型 | 用途 |
|---|---|
| `targetPath` | ファイル／フォルダー種別と絶対パス |
| `stringList` | 制作者、販売者 |
| `titledPathList` | タイトル付き関連文書パス |
| `titledUrlList` | タイトル付き関連URL |
| `assetTypeSet` | 種類IDの配列 |
| `tagSet` | タグIDの配列 |
| `recordStatus` | レコード状態 |

`builtin.acquisitionSource`は`singleSelect`として保存し、カラム定義の選択肢を購入／入手元マスタとして扱う。旧形式のテキスト値は起動時に選択肢IDへトランザクション移行する。

### 18.4 組み込みカラムID

組み込みカラムの固定IDは以下を基本とする。

| 区分 | ID |
|---|---|
| 基本 | `builtin.name`, `builtin.assetTypes`, `builtin.targetPath`, `builtin.tags`, `builtin.status`, `builtin.favorite`, `builtin.description`, `builtin.notes` |
| 更新情報 | `builtin.currentVersion`, `builtin.assetUpdatedDate` |
| 制作者・入手 | `builtin.creators`, `builtin.sellers`, `builtin.acquisitionSource`, `builtin.productUrl`, `builtin.acquiredDate`, `builtin.priceJpy`, `builtin.orderNumber`, `builtin.receiptPath`, `builtin.purchaseAccount` |
| ライセンス判定 | `builtin.creditRequired`, `builtin.linkRequired`, `builtin.logoRequired`, `builtin.commercialUseAllowed`, `builtin.modificationAllowed`, `builtin.redistributionAllowed`, `builtin.adultUseAllowed`, `builtin.generativeAiUseAllowed`, `builtin.licenseUnknown`, `builtin.licenseNeedsReview` |
| ライセンス詳細 | `builtin.creditText`, `builtin.licenseNotes`, `builtin.licenseFilePath`, `builtin.licenseUrl`, `builtin.licenseLastCheckedDate`, `builtin.licenseExpiryDate` |
| 関連情報 | `builtin.relatedDocuments`, `builtin.relatedUrls` |

中央ペインのライセンス要約は保存カラムではなく、ライセンス判定カラムから生成する計算表示とする。

### 18.5 レコードJSON

```json
{
  "schemaVersion": 1,
  "id": "019f0000-0000-7000-8000-000000000000",
  "values": {
    "builtin.name": "素材名",
    "builtin.assetTypes": ["type.bgm"],
    "builtin.targetPath": {
      "kind": "folder",
      "path": "D:\\LocalAssets\\SamplePack"
    },
    "builtin.tags": ["tag.fantasy"],
    "builtin.status": "available",
    "builtin.favorite": true,
    "builtin.currentVersion": "1.2.0",
    "builtin.assetUpdatedDate": "2026-07-19",
    "builtin.creators": ["制作者A", "制作者B"],
    "builtin.acquiredDate": "2026-07-01",
    "builtin.priceJpy": 1500,
    "builtin.creditRequired": true,
    "builtin.commercialUseAllowed": true,
    "builtin.licenseLastCheckedDate": "2026-07-19",
    "builtin.relatedDocuments": [
      {
        "title": "利用規約",
        "path": "D:\\LocalAssets\\SamplePack\\LICENSE.pdf"
      }
    ],
    "builtin.relatedUrls": [
      {
        "title": "商品ページ",
        "url": "https://example.invalid/product"
      }
    ],
    "custom.019f0000-0000-7000-8000-000000000001": "追加情報"
  },
  "createdAt": "2026-07-19T01:00:00Z",
  "updatedAt": "2026-07-19T01:30:00Z"
}
```

レコードファイル名は `<レコードID>.json` とし、JSON内の `id` と一致させる。

### 18.6 空値と既定値

- 値が空のカラムは `values` から省略する。
- 省略された文字列、日付、数値、URL、パス、選択値は未入力として扱う。
- 省略された複数選択と専用リストは空配列として扱う。
- 省略されたチェックボックスは `false` として扱う。
- チェックボックスをオフにした場合は、JSONから対象プロパティを削除する。
- 未定義のカラムIDがレコードに残っている場合は、データ損失を防ぐため値を保持し、UIには表示しない。

### 18.7 対象パス

```json
{
  "kind": "file",
  "path": "D:\\LocalAssets\\font.ttf"
}
```

- `kind` は `file` または `folder` とする。
- 対象が一時的に存在しなくても種別を維持できるよう、種別を明示保存する。
- 対象パスが未入力の場合は `builtin.targetPath` 自体を省略する。

### 18.8 カスタムカラム変更

- 名称変更はカラム定義だけを更新し、レコードJSONは変更しない。
- 表示／非表示変更はカラム定義だけを更新する。
- 型変更前に全レコードの既存値を変換可能か検証する。
- すべて変換可能な場合は、全レコードを原子的に変換する。
- 変換できない値が1件でもある場合は、全レコードから対象カラムの値が削除されることを確認ダイアログで提示する。
- ユーザーが確認した場合は、対象カラムの全値を削除して型を変更する。キャンセルした場合は型も値も変更しない。
- カスタムカラム削除は共通のアンドゥ／リドゥ履歴へ記録し、同一セッション中は定義と全レコードの値を復元または再削除できる。

## 19. JSONの原子的保存とスキーマ移行方式

### 19.1 データルートマニフェスト

データルート直下の `manifest.json` は、データセット全体の形式を識別する。

```json
{
  "schemaVersion": 1,
  "catalogId": "019f0000-0000-7000-8000-000000000010",
  "createdAt": "2026-07-19T01:00:00Z"
}
```

- `schemaVersion` はデータセット全体の形式バージョンとする。
- `catalogId` はUUID Version 7とする。
- 初期版は1データセットだけを扱うが、将来の切り替え機能で同一データセットを識別できるようIDを持たせる。

### 19.2 単一JSONファイルの保存

すべてのJSON更新は、対象ファイルへ直接上書きせず次の順序で行う。

1. 更新後のオブジェクトをメモリ上でJSONへシリアライズする。
2. 対象ファイルと同じディレクトリへ、一意な名前の一時ファイルを書き込む。
3. ファイル内容をディスクへフラッシュする。
4. 一時ファイルを再度読み込み、JSON構文、スキーマ、ID、値の型を検証する。
5. 既存ファイルの更新では、一時ファイルとの置換時にロールバック用ファイルを作成する。
6. 新規ファイルでは、一時ファイルを最終ファイル名へ移動する。
7. 置換後のファイルを確認し、正常ならロールバック用ファイルを削除する。

一時ファイルは必ず対象と同じボリュームへ作成し、同一ファイルシステム内で置換する。

### 19.3 単一ファイル保存の起動時復旧

- 最終ファイルが正常で、一時ファイルやロールバック用ファイルだけが残っている場合は、残存ファイルを削除する。
- 最終ファイルが存在しないか不正で、ロールバック用ファイルが正常な場合はロールバック用ファイルを復元する。
- 最終ファイルとロールバック用ファイルの両方が不正な場合は、自動削除や上書きを行わずエラーとして報告する。

### 19.4 複数ファイル更新

一括編集、カスタムカラム型変更、カスタムカラム削除、スキーマ移行など、複数JSONをまとめて更新する場合は `transactions/<transaction-id>/` を使用する。

1. 変更対象と操作内容を `transaction.json` へ記録する。
2. 全変更後ファイルをトランザクション領域へ書き出す。
3. 全ファイルの構文と値を検証する。
4. トランザクション状態を `prepared` としてディスクへ保存する。
5. 元ファイルをロールバック領域へ保持しながら、変更後ファイルへ順次置換する。
6. 全置換成功後、状態を `committed` にする。
7. ロールバック領域とトランザクション領域を削除する。

起動時に未完了トランザクションが見つかった場合は以下のように処理する。

- `prepared` または置換途中: 全ファイルを更新前へロールバックする。
- `committed`: 更新後ファイルを正として、残った一時データを削除する。
- トランザクション情報自体が不正: 自動変更せず、起動を中止してエラーを表示する。

### 19.5 スキーマ移行

- `manifest.json` と各JSONの `schemaVersion` を確認する。
- 現在より古く、アプリが対応する形式は起動時に自動移行する。
- 移行は複数ファイルトランザクションとして実行する。
- 移行前データはトランザクションのロールバック領域へ保持する。
- 移行成功後に `manifest.json` のバージョンを最後に更新する。
- 移行失敗時は全ファイルを移行前へ戻し、エラーを表示する。
- 旧バージョンのアプリへ戻すダウングレード移行は提供しない。
- アプリが対応していない新しい `schemaVersion` は読み込みも編集も行わず、より新しいアプリが必要であることを表示する。
- 件数が多い場合は進捗を表示し、ファイル置換開始前までキャンセルできるようにする。

### 19.6 不正JSONと不正カラム値

- レコードJSONの構文と必須メタデータを読み取れる場合は、各カラム値をカラム定義に対して検証する。
- 型不一致、不正な日付、不正な選択肢ID、不正な値構造などがあるカラムだけを未入力状態へ初期化する。
- 正常なカラム値とレコード自体は維持し、修復後のレコードを原子的に保存する。
- 修復完了後、レコード名またはレコードID、カラム表示名またはカラムID、破損理由、初期化した内容をダイアログへ一覧表示する。
- 修復内容はエラーログにも記録する。
- JSON構文自体が不正、またはレコードIDなど必須メタデータを安全に読み取れない場合は、カラム単位の修復を行わない。
- カラム単位の修復ができないレコードは一覧から除外し、元ファイルを移動、削除、上書きしない。
- 一覧から除外したこと、対象ファイル、エラー内容をダイアログで明示し、ログにも記録する。
- `manifest.json`、カラム定義、設定など全体解釈に必要なJSONが不正な場合は、データ読み込みを中止する。

## 20. 対象パスの正規化と重複判定方式

### 20.1 入力可能な対象パス

- 対象パスは空欄を許可する。空欄の場合は警告状態とする。
- 値を入力する場合は、Windowsの絶対ローカルパスだけを許可する。
- ファイル選択ダイアログ、フォルダー選択ダイアログ、ドラッグ＆ドロップで入力できる。
- 対象パスの手入力とセルへの貼り付けは許可しない。
- 対象パス以外のファイル／フォルダーカラムは、セルへの貼り付けで入力できる。
- 補助パスの貼り付け直後に存在確認し、存在しない場合は値を保持して警告状態にする。
- 相対パス、URL、UNCパス、ネットワークドライブは拒否する。
- Windowsが `Fixed` と判定するドライブを正式対応とし、`Network`、`Removable`、`CDRom` などは拒否する。
- 新規登録時に実在しないパスは登録できない。
- 登録時に実在する対象から `file`／`folder` 種別を判定して保存する。
- 登録後に対象が移動・削除されて存在しなくなった場合は、保存済みのパスと種別を維持してリンク切れとして扱う。

### 20.2 保存前の正規化

入力された対象パスは、保存前に次の順序で正規化する。

1. コピー時に付く外側の引用符が対になっている場合だけ除去する。ファイル名として有効な空白は除去しない。
2. Windowsのフルパスへ変換する。
3. 区切り文字を `\` に統一する。
4. ドライブルート以外の末尾区切り文字を除去する。
5. ドライブ文字を大文字へ統一する。
6. 正規化後の絶対パスをレコードJSONへ保存する。

環境変数、相対パス、8.3形式の短いパス、シンボリックリンク、ジャンクションは展開しない。

### 20.3 比較キー

- 保存値から比較用のフルパスを生成する。
- `\\?\` プレフィックスの有無、区切り文字、末尾区切り文字、ドライブ文字を比較前に統一する。
- Windowsの一般的なパス比較に合わせ、`OrdinalIgnoreCase`相当の大文字小文字を区別しない比較を行う。
- 比較キーは派生値とし、レコードJSONには保存しない。
- シンボリックリンク、ジャンクション、ハードリンクの実体は解決しない。
- 登録したパスから対象を開けることを要件とし、異なる文字列のパスが同じ実ファイルを指していても、正規化後の比較キーが異なれば別パスとして扱う。

### 20.4 重複判定

- 重複禁止の対象は `builtin.targetPath` だけとする。
- ライセンスファイル、領収書、関連文書などの補助パスは重複を許可する。
- レコードの新規保存と対象パス変更の直前に、全レコードの対象パス比較キーを確認する。
- 自分自身の現在のレコードIDは比較対象から除外する。
- 利用不可、アーカイブ、リンク切れのレコードも重複判定に含める。
- 重複時は保存を中止し、同じ対象パスを持つレコード名とレコードIDを表示する。
- 履歴内の削除済みレコードは重複判定から除外し、その対象パスを直ちに別レコードへ登録できる。
- 削除済みレコードをアンドゥする時点で対象パスが他レコードに使われている場合は、エラーダイアログを表示してアンドゥを実行しない。

### 20.5 拡張子による種類の自動設定

- 種類定義に拡張子の配列を持たせる。
- 拡張子は先頭の `.` を含む小文字へ正規化して保存し、大文字小文字を区別せず比較する。
- 対象ファイル登録時にレコードの種類が空の場合だけ、拡張子と一致する種類を検索して自動設定する。
- 複数の種類に同じ拡張子が設定されている場合は、一致した種類をすべて設定する。
- 既に1件以上の種類が設定されている場合は、自動追加や置換をせず既存値を維持する。
- 対象フォルダー登録時は、拡張子による種類の判定を行わない。
- 拡張子が設定されていないファイルと、拡張子がどの種類にも一致しないファイルは種類を空のままとし、警告を表示しない。
- 種類定義の拡張子設定を変更しても、既存レコードの種類は自動更新しない。

## 21. 起動時パス確認の並行数とキャンセル方法

### 21.1 実行タイミング

- レコードJSONの読み込み完了後にメイン画面を表示する。
- 起動時確認が有効な場合は、メイン画面表示後にバックグラウンドでパス確認を開始する。
- パス確認の完了をメイン画面表示の条件にしない。
- 起動時確認が無効な場合は、自動確認を行わず未確認状態で表示する。
- パスの新規登録または変更時は、そのパスを直ちに確認する。

### 21.2 確認対象

- 表示中の100件だけでなく、読み込んだ全レコードを確認対象にする。
- 対象パスに加え、ライセンスファイル、領収書、関連文書、カスタムファイル／フォルダーカラムも確認対象にする。
- 同じ補助パスが複数レコードに登録されている場合は、1回だけファイルシステムへ問い合わせて結果を共有する。
- 空欄のパスはファイルシステムへ問い合わせず、対象パスだけを未登録警告とする。

### 21.3 確認状態

| 状態 | 意味 |
|---|---|
| `unchecked` | 未確認またはキャンセルで未処理 |
| `checking` | 確認中 |
| `available` | 存在し、アクセスできる |
| `missing` | 存在しない |
| `accessDenied` | 権限不足で確認不能 |
| `error` | その他の入出力エラー |

- 確認結果は再生成可能なセッションキャッシュとし、レコードJSONへ保存しない。
- ファイルかフォルダーかはパス登録時に取得する内部情報であり、ユーザー向けの確認項目や不一致状態として扱わない。
- `missing`、`accessDenied`、`error` は同じ「リンク切れ」にまとめず、詳細画面とダイアログで理由を区別する。

### 21.4 並行処理

- 最大8件を同時確認する固定ワーカープールを使用する。
- 並行数は初期版ではユーザー設定にしない。
- UIスレッドではファイルシステムへ問い合わせない。
- 進捗は `確認済み件数 / 全件数` としてステータスバーへ表示する。
- 個別確認の結果は完了した順に画面へ反映する。

### 21.5 キャンセル

- ステータスバーにキャンセル操作を表示する。
- キャンセル後は新しい確認を開始せず、実行中の確認だけを完了させる。
- 未処理のパスは内部的に `unchecked` のままとし、レコード単位の警告は表示しない。
- キャンセルされたことと未確認件数はステータスバーへ表示する。
- キャンセルしても、確認済み結果は維持する。
- 手動操作では「全レコードを確認」だけを提供する。
- 個別確認操作は提供せず、パスの登録または変更直後に変更対象だけを自動確認する。
- 手動確認を再実行すると、対象範囲の既存キャッシュを更新する。
